# 집회 지원 웹서비스 — 개발 지침서 (Claude Code용)

> 이 문서는 Claude Code가 곧바로 구현에 착수할 수 있도록 작성된 사양서입니다.
> 막히는 지점이 있으면 추측해서 구현하기 전에 **핵심 설계 원칙(2장)** 을 기준으로 판단하세요.
>
> **변경 이력**
> - v1: 초기 4기능 사양.
> - v2: 백엔드 Supabase 확정, 2만+ 트래픽 위해 Cloudflare CDN 캐시 도입(Realtime→캐시 폴링), 신고 자정작용(고마워요/허위에요).
> - v3: 공지 권한 모델 확정(관리자 코드 + 운영자별 개별 게시 코드), 피드백/건의 기능 추가(A안: 인앱 저장 + 관리자 열람).
> - v4: 미결 사항 확정 — snapshot 생성 방식(Cloudflare Worker Pull-through), feedback 보관 7일, 관리자 진입 `/admin`, ZONES 확정(1-1~1-5, 2-1~2-5).

## 0. 한 줄 요약

위치 기반 익명 체크인으로 집회 규모를 파악하고, 필요 물자·안전 경고·공지를 빠르게 공유하는 **모바일 우선 / 로그인 없음 / 개인정보 비수집** 웹 서비스. **2만 명 이상 트래픽을 저비용으로 감당**하는 것이 핵심 제약. 스레드를 대체하는 게 아니라, 스레드가 못 하는 **구조화된 현황판**을 지향한다.

---

## 1. 프로젝트 개요와 맥락

- **목적**: 자발적으로 모인 시민들의 집회를 효율적으로 운영하도록 돕는 경량 웹 도구.
- **포지셔닝**: 토론·잡담은 스레드 등 기존 SNS에 맡기고, 이 서비스는 **"지금 상황이 한눈에 보이는 현황판"** 역할에 특화한다(실시간 집계 규모, 해결되는 물자 체크리스트, 구역별 경고, 권위 있는 공지).
- **사용자**: 스마트폰을 든 일반 참여자(대다수) + 소수의 운영 인력.
- **환경 제약**:
  - 집회 초반 → **속도가 최우선**. 완벽함보다 동작하는 MVP를 먼저.
  - **2만+ 동시 트래픽**을 저비용으로 감당해야 함(3·4·5장).
  - **개인정보 비취급** → 로그인·이름·전화번호·정밀 위치 저장 금지.
  - 현장 네트워크 불안정 가능 → 가벼운 페이지, 폴링 기반, 실패에 관대한 UX.

## 2. 핵심 설계 원칙 (모든 판단의 기준)

1. **개인정보 제로 (Privacy by default)**
   - 계정·로그인 없음. 이름/연락처/이메일 수집 금지.
   - 위치는 항상 **저정밀도로 반올림**(개인 추적 불가능하게).
   - IP 영구 로깅 금지. 모든 데이터 **자동 만료(TTL)**.
   - 피드백·공지 등 새 기능도 **기존 프라이버시 울타리(Supabase) 안에 머물게** 한다. 외부 메일함 등으로 내용을 빼지 않는다.
   - 데이터를 적게 모으는 것이 곧 참여자 보호다.

2. **빠르고 단순하게 (Ship fast)**
   - 빌드 스텝/의존성 최소화. 실시간 동기화는 **폴링** 기반(Realtime 미사용).

3. **저비용·저자원 (2만 대응)**
   - 읽기 트래픽은 **CDN 캐시가 전부 흡수**, 원본(Supabase)은 거의 안 건드림.
   - 이미지 업로드·무거운 에셋 금지.

4. **악용에 강하게 (Abuse-resistant)**
   - 자동 만료 + 레이트 리밋 + 길이 제한 기본 적용.
   - 커뮤니티 투표(고마워요/허위에요)로 자정작용, 단 안전 경고엔 완충 장치(7.3).
   - 행동 유발형 **공지는 권한 코드로 게이트**(7.4).

> 우선순위: **개인정보 보호 > 단순/속도 > 저비용 > 악용 방지**, 단 안전·프라이버시는 항상 우선.

## 3. 기술 스택 (확정)

- **프론트엔드**: 정적 SPA. Vite + React(또는 Preact). 모바일 우선, Tailwind 또는 단일 CSS.
- **읽기 경로(핵심)**: **Cloudflare**(Pages + Worker/Cache). 사용자는 **CDN 캐시된 `/snapshot.json`** 만 폴링. Supabase 직접 호출 안 함.
- **쓰기 + 권한 검증 + 관리자 열람**: **Supabase**(Postgres + Edge Functions). 모든 코드 검증·민감 테이블 접근은 **Edge Function 안에서 service role로만**(클라이언트/anon 키로는 불가).
- **플랜**: 2만 대응 시 **Supabase Pro($25/월)** 권장. 읽기는 CDN이 흡수하므로 Pro 한 장으로 충분.
- **미사용**: Supabase **Realtime 구독**(동시 연결 한도: 무료 200 / Pro 500 → 2만 불가). 대신 캐시 폴링.
- **외부 메일 서비스 불필요**: 피드백은 A안(인앱 저장)이라 메일 API 의존이 없음.

## 4. 시스템 아키텍처

```
[2만+ 모바일 브라우저(참여자)]
   │  익명 세션ID = localStorage 무작위 UUID (신원으로 저장 안 함)
   │
   ├── 읽기(폴링 5~10초) ──> [Cloudflare CDN: /snapshot.json]  (물자+경고+규모+공지)
   │                              │ TTL(5~10초) 만료 시에만 1회 (Pull-through)
   │                              ▼
   │                    [Cloudflare Worker → Supabase Edge Function: 스냅샷 생성/집계]
   │
   └── 쓰기 ──> [Cloudflare Worker(레이트리밋)] ──> [Supabase Edge Function]
                                                        │ (코드 검증/집계, service role)
                                                        ▼
                                                   [Postgres]
                                                     ├─ checkin_counters (구역 카운터, TTL)
                                                     ├─ supplies
                                                     ├─ alerts (고마워요/허위에요 카운터, 짧은 TTL)
                                                     ├─ announcements (게시 코드로 작성)
                                                     ├─ posting_codes (운영자별 게시 코드, 해시)
                                                     └─ feedback (인앱 저장, 관리자만 열람)

[관리자(본인/운영자)] ── 관리자/게시 코드 입력 ──> Edge Function ── service role ──> 위 테이블 관리/열람
[관리자 코드(마스터)] = 환경변수 시크릿(해시). 테이블에 두지 않음.
```

**핵심 원리**
- **읽기/쓰기 분리.** 읽기는 전부 CDN 캐시, 쓰기만 Supabase.
- 민감 테이블(posting_codes/feedback)은 **RLS로 잠그고 Edge Function만 접근.**
- 모든 쓰기에 레이트 리밋, 모든 (실시간성) 레코드에 `expires_at`.

## 5. 규모/성능 요건 (2만+ 대응)

- **읽기**: 단일 `/snapshot.json`(물자 + 활성 경고 + 집계 규모 + **공지**)을 **CDN TTL 5~10초**로 캐싱. 사용자는 이것만 폴링. 스냅샷은 수십 KB 이내로 가볍게.
- **체크인 집계**: 개별 행 count 금지. **구역별 카운터 증가** 후 스냅샷에서 합산.
- **쓰기 부하**: 읽는 사람이 압도적, 쓰는 사람은 일부. 투표·체크인은 **카운터 원자적 증가**.
- **피드백은 읽기 폭주와 무관**: 스냅샷에 들어가지 않음(아무도 폴링 안 함, 관리자만 가끔 열람). 쓰기도 저빈도라 Supabase 부하 미미. 도배 공격은 **Worker 레이트 리밋이 엣지에서 차단**.
- **egress 가드**: 스냅샷에 적절한 `Cache-Control`, 캐시 미스 시에도 원본 1회만. 사용자별 무거운 쿼리 금지.
- **장애 허용**: 쓰기 실패해도 읽기(캐시) 동작. 캐시가 잠깐 낡아도 치명적이지 않음.

## 6. 데이터 모델

> 모든 (실시간성) 테이블에 `id`(uuid), `created_at`, `expires_at`. **신원 식별 컬럼 금지.**

### checkin_counters
| 컬럼 | 설명 |
|---|---|
| geohash | 격자(~100m)로 스냅한 위치. 정밀 좌표 저장 금지 |
| bucket | 시간 버킷(예: 5분) |
| count | 활성 체크인 수(원자적 증가) |
| expires_at | 예: 30분 |

### supplies
| 컬럼 | 설명 |
|---|---|
| item / note / zone | 물자명·메모·구역(길이 제한) |
| status | `needed` / `resolved` |
| helpful_count | "도움됨"(정렬 보조, 선택) |
| expires_at | 예: 6시간 |

### alerts (안전 경고/신고)
| 컬럼 | 설명 |
|---|---|
| zone / description | 위치 + **상황·행동만**(사진·이름 금지) |
| thanks_count | "고마워요" |
| false_count | "허위에요" |
| hidden | 자정 로직으로 숨김 |
| expires_at | 예: 30~60분 |

### announcements (공지)
| 컬럼 | 설명 |
|---|---|
| body | 공지 내용 |
| pinned | 상단 고정 |
| posting_code_id | **어떤 게시 코드로 올렸는지**(최소 추적, 신원 아님) |
| expires_at | 예: 집회 종료 또는 12시간 |

### posting_codes (운영자별 게시 코드)
| 컬럼 | 설명 |
|---|---|
| label | 식별용 태그(예: "운영자1"). **실명 등 개인정보 금지** |
| code_hash | 코드의 **해시**(평문 저장 금지) |
| revoked | 폐기 여부 |
| created_at | |

- **RLS 완전 잠금**: 공개 anon 키로 읽기/쓰기 불가. Edge Function(service role)만 접근.

### feedback (문의·건의)
| 컬럼 | 설명 |
|---|---|
| body | 본문(평문 — 본인이 읽어야 하므로). 길이 제한 |
| handled | 처리 표시(선택) |
| created_at | |
| expires_at | 7일 (무한정 금지) |

- **RLS 완전 잠금**: anon 키로 읽기 불가. 작성은 Edge Function 경유 insert만, 열람은 관리자 코드 게이트.

### 관리자 코드(마스터) — **테이블 아님**
- **환경변수/Edge Function 시크릿에 해시로 저장.** API로 조회 불가. 게시 코드 발급·폐기, 피드백 열람 등 최고 권한 작업만 인가. 본인만 보유.

## 7. 기능별 상세 명세

### 7.1 위치 기반 체크인
- **장소: 올림픽공원 핸드볼경기장 일대(실외).** 야외라 브라우저 GPS 사용 가능.
- "체크인" → Geolocation → **클라이언트에서 반올림/격자 스냅** → 격자·버킷 카운터 +1. 동시에 사용자가 **구역(zone)을 선택**(부록 A의 ZONES). 앱 열린 동안 주기 갱신(예: 5분).
- **두 신호의 역할 분리**: GPS 격자 = 전체 밀집도/히트맵용. **구역 선택 = "어느 게이트/면 앞" 귀속의 주신호**(게이트가 서로 가까워 GPS만으론 구분이 어려우므로). 구역 선택은 필수에 가깝게, GPS는 거부 시에도 구역만으로 체크인 가능.
- 표시: 전체 활성 수 + **구역별 분포**(개별 점 아님, 밀집도만). 지도는 선택. UI에 "추정치" 명시.
- 구역 목록은 **하드코딩하지 말 것** — 부록 A의 단일 config(ZONES)에서 읽어온다.

### 7.2 필요 물자
- 누구나 등록(item/note/zone). `needed` 우선·최신순. "해결됨"→`resolved`(되돌리기 허용).
- 길이 제한, 레이트 리밋, 자동 만료(예: 6h), 스팸 필터.

### 7.3 안전 경고(신고) + 자정작용
- **하드 제약**: 사진/이름 금지, **상황·위치·시각만**, 등록 화면 고정 안내("개인 특정·직접 대응 금지, 위험 시 운영팀/공식 채널"), 짧은 만료(30~60분).
- **고마워요/허위에요**: "고마워요"=신뢰·정렬 반영, "허위에요" 누적 시 흐리게→숨김.
- **완충 장치(중요)**: 즉시 삭제 금지. ① "허위에요" 최소 임계값 + ② "고마워요" 대비 비율 + ③ 등록 후 최소 표시 시간 보장. 중복투표는 localStorage 기반 best-effort, 표 수는 대략치. 카운터 원자적 증가.

### 7.4 공지 (권한 코드 게이트) — **확정**
- **읽기**: 누구나. 스냅샷에 포함, `pinned` 공지는 상단 배너로 강조.
- **쓰기 = 게시 코드 보유자만.** 신원 인증이 아니라 **권한 부여**(코드 = 올릴 자격). 회원가입 없음.
- **두 등급의 코드**:
  - **게시 코드(`posting_codes`)**: 운영자별로 **개별 발급**. 같은 "게시" 등급. 한 개가 새거나 운영자를 빼야 하면 **그 코드만 폐기**(나머지 영향 없음).
  - **관리자 코드(마스터, 환경변수)**: **본인만 보유.** 게시 코드 발급/폐기 권한. 가장 민감하므로 저장하지 말고 작업 시에만 입력.
- **흐름(전부 Edge Function, 서버에서만 검증)**:
  - 공지 게시: (본문 + 게시 코드) → 코드 해시 대조, `revoked=false`면 insert. announcement에 `posting_code_id` 기록(최소 추적).
  - 게시 코드 발급(관리자): (관리자 코드 + 발급 요청) → 관리자 코드 검증 → **무작위 코드 생성(12자 이상 랜덤)** → 해시만 저장 → **평문은 화면에 한 번만 표시** 후 본인이 운영자에게 전달(평문 미저장).
  - 폐기(관리자): (관리자 코드 + code id) → `revoked=true`.
- **보안 필수**: 코드 검증은 **브라우저에서 절대 안 함**. `posting_codes` RLS 잠금. 코드는 해시 저장. 게시·관리자 엔드포인트 레이트 리밋(무차별 대입 방지). 운영자는 게시 코드를 localStorage에 저장해 재입력 생략 가능(폰 분실 시 그 코드만 폐기).
- **런칭 시작**: `posting_codes`에 본인 코드 1개만. 운영자 생기면 추가만.

### 7.5 문의·건의(피드백) — A안 확정
- **목적**: 사이트 전반에 대한 문의/건의를 받기.
- **방식(A안)**: 인앱 텍스트 박스 → Edge Function → **`feedback` 테이블 저장.** 본인이 **관리자 코드로 관리자 화면에서 열람.** 외부 메일함/서비스로 내용이 나가지 않음.
- **단방향·익명.** 답장하려면 신원이 필요해지므로(=PII), MVP는 "받기만".
- **수신지 고정**: 사용자가 받는 주소를 지정하는 입력 없음(스팸 발송 도구 악용 방지). 사용자는 본문만.
- **트래픽**: 스냅샷에 미포함(읽기 폭주 무관). 쓰기는 저빈도 + **Worker 레이트 리밋**으로 도배를 엣지에서 차단.
- **저장 위치 고지**: 내용은 Supabase Postgres의 `feedback`에 **평문** 저장(본인이 읽어야 하므로). 그래서 입력창에 **"민감한 개인정보는 적지 마세요"** 안내. 보관 기간을 정해 무한정 쌓이지 않게.
- **(선택) 알림**: 원하면 "새 피드백 도착" **내용 없는** 알림만 메일로(B 절충). 기본은 인앱 확인.

---

## 8. 개인정보·보안 요건 (필수)

- 로그인/계정/이름/연락처 **비수집**. 위치는 격자 카운터만(정밀 좌표·궤적 금지). IP 영구 저장 금지.
- 모든 실시간성 레코드 TTL + 만료 정리.
- **코드는 해시 저장**(게시 코드). **관리자 코드는 환경변수 시크릿(해시), 테이블 금지.**
- **민감 테이블(posting_codes/feedback) RLS 완전 잠금** — anon 키 접근 불가, Edge Function(service role)만.
- 코드 검증·발급·열람은 **서버(Edge Function)에서만.** Supabase service role 키는 서버 측에만.
- 피드백 본문 평문 → "민감정보 금지" 안내 + 보관 기간 설정.
- 전 구간 HTTPS. 외부 분석/광고 트래커 금지. UI에 프라이버시 고지 1줄.

## 9. 어뷰징·악용 방지 (공통)

- 쓰기 **레이트 리밋**(Cloudflare Worker 단). 입력 **길이 제한** + 스팸/욕설 필터.
- 전 콘텐츠 자동 만료(피드백은 보관 정책).
- 공지: 코드 게이트 + 코드 무차별 대입 방지 레이트 리밋.
- 경고: 고마워요/허위에요 자정 + 완충 장치.
- 읽기 전용 폴백(쓰기 장애 시에도 캐시 읽기 동작).

## 10. API 개요

> 읽기는 캐시 스냅샷 1개. 쓰기/관리자 작업은 Edge Function.

- `GET  /snapshot.json` — **CDN 캐시(TTL 5~10초)**. 물자 + 활성 경고 + 집계 규모 + 공지. 사용자는 이것만 폴링.
- `POST /checkin` — { geohash } → 카운터 +1
- `POST /supplies` / `PATCH /supplies/:id` (status)
- `POST /alerts` / `POST /alerts/:id/thanks` / `POST /alerts/:id/false`
- `POST /announcements` — { body, pinned } + **게시 코드**(서버 검증)
- `POST /admin/codes` — **관리자 코드** → 새 게시 코드 발급(평문 1회 반환)
- `POST /admin/codes/:id/revoke` — **관리자 코드** → 폐기
- `POST /feedback` — { body } (Worker 레이트 리밋, 스냅샷 미포함)
- `GET  /admin/feedback` — **관리자 코드** → 피드백 열람(service role)

모든 쓰기: 레이트 리밋 + 길이/내용 검증 + (실시간성) `expires_at` + 카운터 원자적 증가.

## 11. 프론트엔드 화면 구성 (모바일 우선)

- **단일 페이지 + 탭 4개**: ① 현황(체크인/규모) ② 물자 ③ 경고 ④ 공지
- 상단 **고정 공지 배너**(pinned 있을 때).
- **건의 보내기**: 어디서나 접근 가능한 버튼/모달(본문 입력 + "민감정보 금지" 안내).
- **관리자 화면(`/admin`)**: 관리자/게시 코드 입력 → 공지 작성, 게시 코드 발급/폐기, 피드백 열람. 관리자 코드는 저장하지 않음(매번 입력), 게시 코드는 localStorage 저장 가능.
- 경고 카드에 [고마워요]/[허위에요], 투표 후 낙관적 업데이트.
- 큰 탭·큰 글씨·한 손 조작. 느린 네트워크 대비 로딩/에러 명확. 캐시 폴링(5~10초). 오프라인 시 마지막 데이터 표시. 한국어 UI, 접근성 고려.

## 12. 배포 및 운영

- **프론트 + 읽기 캐시**: Cloudflare Pages + Cache/Worker(무료~저비용). egress를 엣지가 흡수.
- **쓰기/DB/권한**: Supabase Pro($25/월). Edge Function으로 검증·집계·열람.
- **환경변수**: 관리자 코드 해시, Supabase service role 키(서버 전용). (B 절충 쓸 경우에만 알림 메일용 고정 주소.)
- 사용량(요청·DB·egress) 모니터링. 만료 정리로 용량 억제.
- 짧은 도메인 + 현장 배포용 QR.

## 13. 개발 단계 (우선순위)

**MVP (동작 우선)**
1. 정적 SPA + 탭 + Cloudflare 배포
2. `/snapshot.json` 캐시 파이프라인(읽기 경로) — 2만 대응 핵심
3. 물자 등록/해결
4. 체크인(격자 카운터) + 활성 수 표시
5. 경고 등록 + 고마워요/허위에요 자정(완충 장치)
6. 공지: posting_codes/관리자 코드 + 게시/발급/폐기 + 배너
7. 피드백: 입력 + feedback 저장 + 관리자 열람

**2차 확장**
- 구역별 히트맵, 자정 로직 튜닝, 다국어, (원하면) 알림 메일.

## 부록 A. 구역(zone) 설정

> 위치 기능의 유일한 "장소 의존" 부분. **여기 한 곳만 고치면** 체크인·물자·경고의 구역이 모두 바뀐다. 코드 곳곳에 게이트 이름을 하드코딩하지 말 것.

- **장소**: 서울 올림픽공원 핸드볼경기장(SK올림픽핸드볼경기장) **일대, 실외**. 집회는 건물 밖에서 진행.
- **건물 구조**: 출입구는 1층 5개(1-1~1-5), 2층 5개(2-1~2-5), 총 10개. 각 출입구가 독립 spot.
- **구현**: 프론트엔드와 스냅샷 집계가 공유하는 **단일 config 배열 `ZONES`** 로 정의. 각 zone은 `{ id, label, group }` 정도면 충분.

```js
// config/zones.js
export const ZONES = [
  { id: "g1-1", label: "1-1 출입구", group: "1층" },
  { id: "g1-2", label: "1-2 출입구", group: "1층" },
  { id: "g1-3", label: "1-3 출입구", group: "1층" },
  { id: "g1-4", label: "1-4 출입구", group: "1층" },
  { id: "g1-5", label: "1-5 출입구", group: "1층" },
  { id: "g2-1", label: "2-1 출입구", group: "2층" },
  { id: "g2-2", label: "2-2 출입구", group: "2층" },
  { id: "g2-3", label: "2-3 출입구", group: "2층" },
  { id: "g2-4", label: "2-4 출입구", group: "2층" },
  { id: "g2-5", label: "2-5 출입구", group: "2층" },
  { id: "etc",  label: "기타 / 모름", group: "기타" },
];
```

- **`기타/모름`은 항상 둔다**: 위치를 모르거나 GPS 거부 시 선택지가 있어야 체크인이 막히지 않음.
- **geohash 정밀도**: 야외 밀집도 표시는 ~100m 격자면 충분(개인 추적 방지 + 분포 해상도 균형). 게이트 단위 구분은 GPS가 아니라 위 `ZONES` 수동 선택으로 처리.
- **운영 중 조정**: 인원이 특정 면으로 쏠리면 zone을 쪼개거나 추가만 하면 됨(목록 수정 → 재배포).
